Fase 1: Dominio Rico y Arquitectura Limpia

Descripción General
El objetivo de este trabajo práctico es desarrollar el núcleo lógico de un Sistema de Gestión de Expedientes
(SGE). Este sistema posibilita la organización y administración efectiva de la información, donde cada
expediente contendrá un conjunto de datos propios junto con una colección de trámites asociados.
Esta primera fase se construirá utilizando los principios de Arquitectura Limpia y tácticas de Domain-Driven
Design (DDD) vistas en la teoría. El objetivo es lograr un código altamente desacoplado, testeable y
preparado para evolucionar. En una entrega futura este mismo núcleo de negocio se conectará a una base
de datos SQLite y a una interfaz web (Blazor o Web API) sin necesidad de modificar las reglas de negocio.

1. Estructura de la Solución
La solución deberá llamarse SGE y deberá contener estrictamente los siguientes cuatro proyectos,
respetando la Regla de Dependencia:
1.​ SGE.Dominio (Class Library - No depende de nadie)
2.​ SGE.Aplicacion (Class Library - Depende de SGE.Dominio)
3.​ SGE.Infraestructura (Class Library - Depende de SGE.Aplicacion y SGE.Dominio)
4.​ SGE.Consola (Console App - Depende de todos para actuar como Composition Root, es decir, el
punto central donde se crean las instancias de las clases proveyéndoles las dependencias necesarias)

1.1 Organización Interna (Feature Folders)
Dentro de cada proyecto (especialmente en Dominio y Aplicación), se recomienda adoptar la organización
por Módulos de Negocio (Feature Folders) en lugar de la clásica agrupación por "tipo técnico" (se debe
evitar crear carpetas genéricas llamadas Entidades, DTOs o CasosDeUso).
Para los nombres de estas carpetas utilizaremos el plural (Expedientes, Tramites), mientras que las
clases en su interior irán en singular (Expediente, Tramite).

💡 Nota Técnica (C#): ¿Por qué la carpeta en plural y la clase en singular? Microsoft aconseja

firmemente que un espacio de nombres (namespace, que se genera a partir del nombre de la
carpeta) no se llame exactamente igual que un tipo o clase en su interior. Si la carpeta y la clase
se llamaran ambas Tramite, el compilador de C# se confundiría al intentar resolver las
referencias y generaría errores. Al usar el plural para la carpeta, evitamos esta colisión.

💡 Nota sobre la Convención de Nomenclatura: Notarán a lo largo del TP una mezcla

Dominio en Español y Roles Técnicos en Inglés. Utilizamos el español para reflejar el Lenguaje

Lógica de Negocio: Gestión del Estado y Modificación del Expediente
El ciclo de vida de un expediente implica distintos tipos de alteraciones. Para respetar el diseño orientado al
dominio, la entidad Expediente debe proveer métodos que reflejen la intención real del negocio.
A. Modificar Carátula: Si el usuario cometió un error al ingresar el expediente, debe poder corregir la
carátula. Esta es una operación excepcional que no afecta el flujo de vida del expediente.
●​ Directiva: La entidad Expediente debe exponer un método (ej: public void
ModificarCaratula(Caratula nuevaCaratula, Guid idUsuario)) que actualice este

dato, el usuario responsable y la fecha de modificación.
B. Cambio de Estado Automático (Por trámites): Al agregar, modificar o eliminar un trámite, el sistema
puede cambiar automáticamente el estado del expediente según la etiqueta del trámite que haya quedado
como "último" (considerando como "último" a aquel trámite con la fecha de creación más reciente):
●​ Si la etiqueta es "Resolución" -> El estado cambia a "Con resolución".
●​ Si la etiqueta es "Pase a estudio" -> El estado cambia a "Para resolver".
●​ Si la etiqueta es "Pase al Archivo" -> El estado cambia a "Finalizado".
●​ Si el expediente se queda sin trámites (ej: se eliminó el único que tenía) -> El estado vuelve a
"RecienIniciado".
●​ Directiva: La entidad Expediente debe ser la única responsable de evaluar y aplicar esta regla para
mutar su estado. Para lograrlo, sugerimos que exponga un método (ej: public bool
ActualizarEstado(EtiquetaTramite? ultimaEtiqueta, Guid idUsuario)) que
efectúe el cambio. Sugerimos que retorne true si el estado efectivamente cambió (esta información
será muy útil para la orquestación en la capa de Aplicación, que utilizará este valor para saber si
debe ordenar al repositorio que persista la modificación, optimizando así las llamadas a la base de
datos) y que reciba el idUsuario porque al mutar la entidad, se debe registrar quién causó esta
actualización automática. (Nota de Diseño: Esta estrategia específica en la firma del método
—retornar un booleano para optimizar la persistencia, o utilizar un ultimaEtiqueta en nulo para
representar expedientes que se han quedado sin trámites— es una sugerencia para guiar su diseño.
Tienen total libertad para resolver este escenario con otros métodos o parámetros, siempre y cuando

respeten la directiva principal: usar un Modelo Rico donde la propia entidad concentre la inteligencia
y decida sobre su estado. Al recibir solo la etiqueta en lugar del objeto trámite completo,
mantenemos las entidades desacopladas).
C. Cambio de Estado Manual (Flujo natural): El usuario debe poder cambiar el estado de un expediente
manualmente, independientemente de cuál sea su último trámite.
●​ Ejemplo de negocio: Si el último trámite es una "Resolución", el sistema lo pasa automáticamente a
"Con resolución". Sin embargo, más adelante, el usuario debe poder editar el expediente para
pasarlo al estado "En notificación" sin necesidad de agregar ningún trámite.
●​ Directiva: La entidad Expediente debe exponer un método (ej: public void
CambiarEstado(EstadoExpediente nuevoEstado, Guid idUsuario)) que reciba el
nuevo estado y actualice la información correspondiente.

💡 Consejo de Diseño (Firmas de los Repositorios): Sugerimos que los repositorios contengan

métodos base como Agregar, ObtenerPorId, Modificar y Eliminar, además de
cualquier otro método de consulta que se consideren necesario para sus Casos de Uso (ej.
ObtenerTodos para expedientes u ObtenerPorExpedienteId para trámites). Las
operaciones de mutación (Agregar, Modificar, Eliminar) deben retornar void. En
Arquitectura Limpia, si el repositorio no encuentra el elemento para modificar o eliminar, no
debe retornar un booleano false, sino lanzar una excepción (ej. RepositorioException),
ya que intentar operar sobre algo que no existe es un comportamiento anómalo. Por el
contrario, los métodos de consulta como ObtenerPorId deben estar diseñados para retornar
la entidad o null (ej: Expediente?) en caso de no hallarla. Esta no es una decisión arbitraria:
muchas veces un Caso de Uso necesita consultar si una entidad existe (por ejemplo, para evitar
crear un duplicado) y actuar en consecuencia. Si el repositorio lanzara una excepción al no
encontrarla, nos obligaría a programar el flujo de la aplicación usando bloques try/catch, lo

cual es una mala práctica conocida como Exception Driven Development (Desarrollo Guiado por
Excepciones). Será responsabilidad del Caso de Uso evaluar ese null y decidir si debe lanzar
una excepción de negocio (ej. EntidadNoEncontradaException).

💡 Consejo de Diseño (Tipos de Retorno): Al definir los métodos de los repositorios que

devuelven colecciones de objetos (ej. "Obtener todos los expedientes"), es común usar
List<Expediente>. Sin embargo, para diseñar interfaces verdaderamente resilientes y
preparadas para la Fase 2 (cuando nos conectemos a bases de datos), los animamos a leer el
Apéndice A al final de este documento, donde se explica por qué es preferible usar
IEnumerable<T>.

Casos de Uso y DTOs
Cada Caso de Uso será una clase independiente (utilizando el sufijo UseCase, ej.
AgregarExpedienteUseCase) con un método Ejecutar. La comunicación de entrada y salida se hará
estrictamente mediante records DTOs, utilizando los sufijos Request y Response (ej.
AgregarExpedienteResponse Ejecutar(AgregarExpedienteRequest request)). Devolver un
DTO en lugar de datos primitivos (como un Guid) permite que la firma del método sea escalable a futuro.
Se deben implementar, como mínimo, los siguientes Casos de Uso:
1.​ Alta, Baja y Modificación de Expedientes.
○​ Aclaración sobre la Modificación: Como la entidad separa claramente la intención de corregir
la carátula (una operación excepcional) y cambiar el estado manual (parte del ciclo de vida
ordinario), se deben crear dos Casos de Uso totalmente separados:
ModificarCaratulaExpedienteUseCase y CambiarEstadoExpedienteUseCase.
○​ Aclaración sobre la Baja (Eliminación en Cascada): Como regla fundamental de negocio, al
eliminar un expediente, se deben eliminar todos sus trámites asociados. El Caso de Uso
encargado de la baja deberá coordinar esta acción, asegurándose de eliminar primero los
trámites desde su repositorio, y luego el expediente desde el suyo (este orden es vital para
evitar errores de integridad referencial a futuro).
2.​ Alta, Baja y Modificación de Trámites.
○​ Aclaración sobre el Cambio de Estado Automático: Como estas tres operaciones pueden
generar la necesidad de actualizar el estado del expediente (alterando cuál es el "último"
trámite o cambiando la etiqueta del último trámite), la lógica de orquestación para actualizar
el estado es idéntica en los tres. Para evitar repetir código (Principio DRY "Don't Repeat
Yourself"), se deberá crear un Servicio de Aplicación (ej.
ActualizacionEstadoExpedienteService) e inyectarlo en estos tres Casos de Uso.
○​ La responsabilidad del Servicio: Este servicio actuará como orquestador. Recuperará el
expediente, buscará todos sus trámites para determinar cuál es el "último", extraerá la
etiqueta de dicho trámite y le pedirá a la entidad que evalúe su nuevo estado invocando:

bool cambio = expediente.ActualizarEstado(ultimaEtiqueta, idUsuario).

El servicio solo llamará a _expedienteRepository.Modificar(expediente) si el
método retornó true.

💡 Nota de Diseño: El poder del Modelo Rico Noten que el Servicio de Aplicación no decide

cuál es el nuevo estado ni modifica la propiedad directamente. Toda la lógica sigue residiendo
dentro de la propia entidad Expediente.
●​ Encapsulamiento y Seguridad: Al usar { get; private set; } en la propiedad
Estado, impedimos que cualquier componente del sistema cambie el estado
accidentalmente saltándose las reglas. El compilador nos obliga a pasar siempre por el
embudo de validación.
●​ Testeabilidad: Probar que las reglas de cambio de estado funcionan requiere apenas
escribir un Test Unitario rapidísimo instanciando un Expediente y llamando a su
método. No hace falta crear mocks de repositorios.

💡 NOTA AVANZADA DE DISEÑO: El "Tiempo" en DDD (Opcional) En un modelo DDD purista, las

Entidades nunca deben acceder directamente al reloj del sistema (por ejemplo, llamando a
DateTime.Now). El reloj de la computadora se considera un detalle que pertenece a la capa de
Infraestructura. Además, generar la fecha internamente dificulta muchísimo la creación de Tests
Unitarios, ya que el valor temporal cambiará en cada ejecución de la prueba, haciendo casi imposible
verificar el estado final con precisión (un Assert).
La solución ideal: La fecha y hora exactas de creación o modificación deberían llegar a la Entidad por
parámetro en cada método (ej: ModificarCaratula(..., DateTime fechaModificacion)).
Generalmente, esta fecha es obtenida en la capa de Aplicación a través de un servicio de
infraestructura inyectado (como un ITimeProvider) y luego "pasada hacia abajo" al Dominio.
Para este Trabajo Práctico: Con el objetivo de no complicar el enunciado sumando más abstracciones,
hemos flexibilizado este requerimiento. Es totalmente válido y aceptado que utilicen
DateTime.Now directamente dentro de sus Entidades para actualizar las fechas de forma interna. Sin
embargo, si alguno desea implementar la solución purista pasando las fechas por parámetro a los
métodos del Dominio, ¡es la mejor manera de hacerlo y los animamos a intentarlo!

APÉNDICE A: Colecciones en Interfaces (List<T> vs IEnumerable<T>)
Cuando diseñamos los contratos (interfaces) de nuestros repositorios en la Capa de Aplicación, usar
List<T> como valor de retorno es lo más intuitivo. Sin embargo, en arquitecturas profesionales, utilizar
una interfaz más restrictiva como IEnumerable<T> trae enormes beneficios de diseño. Cabe aclarar que
IEnumerable<T> es una interfaz fundamental en la plataforma .NET y la veremos en detalle más adelante
en el curso. Aún así, animamos a utilizarla en los métodos de los repositorios que devuelven una lista de
entidades.
Las tres razones principales para aplicar esta práctica son:

1. Memoria y Ejecución Diferida (Deferred Execution)
Un List<T> es una estructura de datos materializada 100% en la memoria RAM. Si en el futuro la base de
datos tiene 100.000 trámites, devolver un List<T> obligará al servidor a reservar memoria para 100.000
objetos de golpe, saturando el sistema.
●​ La ventaja de IEnumerable<T>: Esta interfaz no representa "una lista cargada en memoria", sino
una promesa de entregar elementos de a uno. Cuando nos conectemos a SQLite mediante un ORM
(como Entity Framework), devolver un IEnumerable<T> permitirá que los datos viajen desde la
base de datos a medida que el Caso de Uso los recorre en un foreach (como si fuera un stream o
flujo de agua), consumiendo poquísima RAM.

2. Encapsulamiento y Protección de Datos (La "Falsa Mutabilidad")
Si la interfaz del repositorio devuelve un List<Expediente>, esa clase trae incorporados métodos
públicos peligrosos como .Add(), .Remove() o .Clear(). ¿Qué pasa si un programador en la Capa de
Aplicación intenta lo siguiente?
var expedientes = _repositorio.ObtenerTodos();
expedientes.Add(nuevoExpediente); // ¡Falsa mutabilidad!

Ese código modificará la lista en la memoria RAM temporal, pero no guardará los cambios en el TXT o Base
de Datos (porque nunca llamó a _expedienteRepository.Agregar()).
●​ La solución: Al devolver IEnumerable<T>, el compilador ocultará los métodos de escritura. Le
resultará imposible al programador hacer un .Add() accidental, obligándolo a utilizar las vías
correctas del repositorio.

3. Flexibilidad para la Infraestructura (Principio de Inversión de Dependencias)
Si la interfaz exige devolver un List<T>, estamos obligando a la capa de Infraestructura a instanciar

●​ Como prácticamente todas las colecciones de C# (Arrays, Lists, HashSets) implementan por defecto
IEnumerable<T>, usar esta interfaz en el contrato le da libertad total al Repositorio para trabajar
con la estructura de datos que le resulte más rápida o eficiente internamente, sin romper el contrato
de la Aplicación.
Devolución de IEnumerable<T> a partir de List<T>
Si un método declara IEnumerable<T> como tipo de retorno, pero internamente se utiliza un List<T>,
puede retornarse la instancia de List<T> sin conversión adicional. Esto es posible porque List<T>
implementa IEnumerable<T>. El compilador resuelve la conversión implícitamente.

public IEnumerable<Expediente> ObtenerTodos()
{
// 1. Creo una lista normal
List<Expediente> miLista = new List<Expediente>();

// 2. La lleno leyendo del archivo TXT...

// 3. ¡Simplemente retorno la lista!
// El compilador la convertirá implícitamente a IEnumerable hacia afuera.
return miLista;
}
